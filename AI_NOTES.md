# AI_NOTES.md

## Tools used
- **AI Orchestration Framework**: Antigravity agentic coding assistant.
- **AI Models**: Gemini models and Claude 3.5 / Opus 4.5 models were used in tandem to generate code, build structures, and plan components.
- **Human-AI Split**: Approximately **85%** of the code was generated by the AI agent based on specifications and context files, with **15%** of the code written, debugged, and configured by the human developer to resolve build issues, handle deployment configuration, and adjust database pooling settings.

## Key decisions made by the developer

### 1. Database-Backed Job Table instead of Redis Queue
- **Decision**: Persist events to a PostgreSQL database table (`Event`) and process them asynchronously rather than using an in-memory queue like Redis (e.g., BullMQ) or a cloud queue like SQS.
- **Reasoning**: To remain within the free tiers of deployment providers without adding external infrastructure dependencies (such as Upstash Redis). Using a PostgreSQL database table simplifies scaffold deployment on Neon/Supabase and ensures durability and transaction-safe logging for audit logs (`ActionLog`).

### 2. Immediate Background Worker Triggering via Next.js `after()` API
- **Decision**: Trigger the background job worker asynchronously immediately inside the webhook route using Next.js 16's standard `after()` callback, alongside a fallback 1-minute Vercel Cron.
- **Reasoning**: This provides near real-time automation feedback on the dashboard and on GitHub when events arrive, while ensuring that the webhook endpoint returns a `200 OK` fast (within milliseconds) without blocking the GitHub server or hitting Vercel's execution time limits.

### 3. Storing OAuth Access Tokens in the Database
- **Decision**: Store the encrypted/plain GitHub OAuth access token inside the `User` table upon login.
- **Reasoning**: NextAuth sessions are stateless JWTs, but because actions (labeling, commenting) are executed in a decoupled, asynchronous background worker (triggered by a cron job or `after()` callback outside of the user's active session context), the worker must have access to the user's API token to instantiate the Octokit client and interact with the GitHub API.

## Hardest bug / wrong turn

### Prisma Connection Failure on Vercel due to Connection Pooling
- **What went wrong**: During the initial deployment of the bot to Vercel, Prisma database queries began failing with connection limit exhaustion and timeout errors when pointing directly to the Supabase PostgreSQL transaction pooling database URL.
- **How it was noticed**: The application failed to connect or perform schema migrations during deployment, and dashboard API queries timed out with database connection errors.
- **How it was fixed**: The developer adjusted the connection strategy by configuring both `DATABASE_URL` (using transaction pooling for runtime queries) and `DIRECT_URL` (bypassing the pooler to run migrations directly) inside the `schema.prisma` file and Vercel environment variables, transitioning Prisma ORM to use connection-pool-aware query strategies.
